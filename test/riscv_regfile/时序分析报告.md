# RISC-V 寄存器组时序分析报告

## 1. 目标

本报告分析当前 `test/riscv_regfile/` 中寄存器组模型的时序行为，重点回答：

- 当前三段式拆分在一个周期内是如何推进的
- 输入、写回、读口输出分别发生在什么顺序
- 当前模型更接近哪一种寄存器组语义
- 对后续探索流水线仿真有什么启发

当前分析对象：

- [RiscvRegFileKernel.hpp](./RiscvRegFileKernel.hpp)
- [main.cpp](./main.cpp)

## 2. 当前结构

当前寄存器组被拆成三个 `KernelComponent`：

1. `RegFileInputStage`
2. `RegFileAccessStage`
3. `RegFileReadStage`

其中：

- `InputStage`
  - 从 simulator 提供的外部输入端口采样：
    - `rs1_addr`
    - `rs2_addr`
    - `rd_addr`
    - `rd_wdata`
    - `rd_we`
  - 再通过输出端口把这些采样值传给 `AccessStage`

- `AccessStage`
  - 独占持有真正的寄存器存储体
  - 当前存储体是二维 `4x8` 的 `StateArray<std::uint64_t>`
  - 先处理写回，再计算两个读口值
  - 再通过输出端口把：
    - `rs1_rdata`
    - `rs2_rdata`
    传给 `ReadStage`

- `ReadStage`
  - 只负责接收 `AccessStage` 的两个读结果
  - 再转发到 kernel 顶层输出

## 3. 当前底层执行顺序

在 XS 当前执行模型里：

- `CycleSimulator::step()` 先推进 simulator 自己
- 再按 `kernels_` 的注册顺序逐个调用 `kernel->run(cycle)`
- `Kernel::run_single()` 默认按 `components_` 的注册顺序逐个调用 `component->run(cycle)`

而每个 `KernelComponent::run(cycle)` 的固定顺序是：

1. `sync_inputs()`
2. `run_single(cycle)`
3. `emit_outputs()`
4. `after_outputs_emitted(cycle)`
5. `advance_phase()`

因此，对于当前寄存器组 kernel，一拍内部的真实顺序是：

1. `InputStage.sync_inputs()`
2. `InputStage.run_single()`
3. `InputStage.emit_outputs()`

4. `AccessStage.sync_inputs()`
5. `AccessStage.run_single()`
6. `AccessStage.emit_outputs()`

7. `ReadStage.sync_inputs()`
8. `ReadStage.run_single()`
9. `ReadStage.emit_outputs()`

10. `RiscvRegFileKernel` 顶层把 `ReadStage` 的结果转发到自己的输出状态
11. `Kernel.emit_outputs()`

## 4. 单拍时序含义

由于这三段之间当前全部使用的是 `wire` 语义端口，并且三个 component 在同一个 kernel 周期内按顺序执行，因此：

- `InputStage` 本拍采样到的地址/写回控制
- 会在 **同一拍** 被 `AccessStage` 看到

- `AccessStage` 本拍算出的 `rs1_rdata/rs2_rdata`
- 会在 **同一拍** 被 `ReadStage` 看到

- `ReadStage` 本拍转发的最终读口值
- 会在 **同一拍** 被顶层 kernel 发射出去

所以当前三段式虽然结构上被拆成了三段，
但时序上它仍然是：

**单周期内串行推进的组合化三段**

而不是：

**跨三个周期逐段流动的真正流水线**

## 5. 写回与读口的相对顺序

`RegFileAccessStage::run_single()` 的顺序是：

1. 先强制 `zero/x0 = 0`
2. 若 `rd_we == true` 且 `rd_addr != 0`，执行写回
3. 再次强制 `zero/x0 = 0`
4. 再根据 `rs1_addr/rs2_addr` 读取寄存器值

因此当前寄存器组的读写语义是：

### 同拍写先于同拍读

也就是说，如果某拍中：

- `rd_addr == rs1_addr`
- 且 `rd_we == true`

那么 `rs1_rdata` 读到的是：

- **本拍新写入后的值**

而不是旧值。

这更接近一种：

- “同步写 + 同拍读返回新值”

的周期级抽象。

## 6. x0 / zero 的语义

当前实现里，无论输入如何驱动：

- 只要写回地址是 `x0`
- 写入都会被覆盖掉

因为访问段每拍都会在写前和写后都执行：

- `regs_.at_name("zero") = 0`

所以标准 RISC-V 约束：

- `x0 == 0`

在当前模型中是被严格保持的。

这也是 `main.cpp` 里第 3 拍专门写 `zero/x0` 的原因。

## 7. 二维寄存器阵列映射

当前寄存器阵列是二维 `4x8`：

- 行号：`index / 8`
- 列号：`index % 8`

即：

- `x0 ~ x7`   在第 0 行
- `x8 ~ x15`  在第 1 行
- `x16 ~ x23` 在第 2 行
- `x24 ~ x31` 在第 3 行

这只是存储体布局变化，
逻辑上的寄存器编号和 ABI 名不变。

## 8. 目前模型的准确定位

当前 `riscv_regfile` 更准确的说法是：

- **阶段信号完全通过端口传递**
- **寄存器存储体只属于访问段内部**
- **单拍内按顺序串行执行三段**

因此它适合：

- 观察单拍内部阶段依赖
- 验证 wire 语义下的阶段拆分是否正确
- 作为后续探索流水线仿真的起点

但它还不等于：

- 真正跨拍流水的寄存器组流水线模型

## 9. 如果要继续探索流水线仿真

后续有两个自然方向。

### 方向 A：保留当前单拍三段，只研究阶段依赖

适合：

- 研究结构拆分
- 研究 validate 规则
- 研究自动生成模块形态

特点：

- 最简单
- 可读性强
- 对当前端口语义最友好

### 方向 B：把阶段边界改成跨拍

如果想真正模拟流水线，就要在阶段之间引入跨拍边界，例如：

- `InputStage -> AccessStage` 之间改成 `reg`
- `AccessStage -> ReadStage` 之间也改成 `reg`

这样就会变成：

- 第 N 拍采样输入
- 第 N+1 拍执行寄存器访问
- 第 N+2 拍给出最终读口结果

这时才更像严格意义上的“流水线仿真”。

## 10. 结论

当前寄存器组模型的核心时序结论是：

1. 三段结构虽然已经拆开，但仍然在同一拍内顺序完成
2. 所有阶段间信号都通过端口传递，没有再共享控制状态
3. `AccessStage` 内部采用“写后再读”的顺序，因此同拍读到的是新值
4. `x0/zero` 在当前模型中始终被强制为 0
5. 二维 `4x8` 存储体当前只改变存储布局，不改变逻辑寄存器语义

因此，当前实现已经非常适合继续往“真正跨拍的流水线寄存器组”方向演化。
