# XS 后续发展建议

有效时间：
- 起始时间：2026-05-13
- 适用阶段：当前 XS 仍处于抽象层快速演进、测试同时承担验证与生成链路职责的阶段

## 当前项目定位

当前项目里的 `test/` 不只是传统意义上的“测试样例”，而是具有双重作用。

### 1. 验证作用

作为手写的周期级仿真案例，用来验证：

- 状态抽象是否合理
- 端口连接是否正确
- 执行顺序是否满足预期
- 标准模块行为是否符合目标

### 2. 生成作用

它还承担未来生成链路的承载体职责，至少有两个方向：

#### 导入方向

从已实现的 HLS 代码以及 Vivado/XO 产物出发，
通过 Python 解析脚本自动生成一个对应的周期级仿真器。

#### 导出方向

从当前 XS 抽象继续生成：

- HLS 代码
- 更低层的 Verilog / SystemVerilog

所以当前 `test/` 更准确地说，是：

- 可执行样例
- 中间建模样例
- 自动导入/导出链路的落脚点

这会直接影响发展顺序：

- 不能把 test 只当临时脚手架
- 不能过早把 test 写死成只服务某一种后端
- 需要把 test 当成“标准模块建模模板”

## 层次关系说明

这些层不是简单“自上而下线性执行一遍”的关系，更准确地说，是：

- 有一条主路径
- 也有若干横向支撑层

### 主路径

当前更适合按下面的主路径理解：

1. 编译期静态层  
   先做语言和模板级别能提前发现的检查。

2. Elaboration / 构造层  
   真正构建对象图、注册状态、连接端口、拼装 kernel/component。

3. Validate 层  
   在真正 `run()` 或导出前做整体合法性检查。

4. 后端层  
   导入 HLS/XO 结果，或导出到 HLS / Verilog。

### 横向支撑层

另外两层更像横向支撑，不是简单插在主路径中间顺着跑：

- 数据 / 配置 / 导入描述层  
  为 elaboration 和后端两边提供结构描述与配置来源。

- 静态分析层  
  在构建系统旁路运行，辅助发现更早期的问题。

## test 在哪里

`test/` 不应该被看成“夹在某两层中间的临时测试目录”，更合适的定位是：

### test 是主路径中间的“标准建模模板层”

它横跨：

- Elaboration / 构造层
- Validate 层
- 后端层

也就是：

- 向上：它依赖当前抽象层的建模能力
- 向下：它又是未来导入/导出链路落地时最接近真实模块形态的模板

可以把它理解成：

- 位于“构造完成之后、后端实现之前”的标准模型样板层

更直白一点说：

- `test/` 不是单纯写给 `make test` 用的
- 而是当前 XS 里最接近“中间表示实例化样本”的地方

所以 test 的位置，应该理解为：

`编译期静态层 -> elaboration/构造层 -> test(标准建模模板层) -> validate 层 -> 导入/导出后端层`

这里的箭头不是严格运行顺序，而是“关注重点从左到右逐步下沉”的关系。

## 各层分工与当前进展

这一节分成两部分来看：

- 这一层理论上负责什么
- 当前 XS 实际已经做到什么

### 1. 编译期静态层

#### 分工

负责处理“编译器在编译模板和常量表达式时就能知道”的问题。

典型职责：

- 模板参数是否合法
- 某些固定 shape 是否合理
- 某些静态禁止组合是否成立
- 某些固定表是否满足静态约束

#### 当前已做到

当前只做到少量。

主要特点：

- 还没有系统性的编译期名字检查
- 还没有系统性的编译期 shape 检查框架
- 主要仍依赖普通模板类型系统和少量静态约束

也就是说，这一层目前还不是主力层。

### 2. Elaboration / 构造层

#### 分工

负责把对象图真正构建出来。

典型职责：

- 构造状态对象
- 注册到 `StateSet` / `StateArrayRegistry`
- 建端口
- 连端口
- 拼 `KernelComponent`
- 拼 `Kernel`
- 拼 `CycleSimulator`
- clone / fullcopy / remove

这一层的重点是：

- “对象能不能被正确构出来”
- “构出来的结构本身是否自洽”

#### 当前已做到

这是当前 XS 已经做得最多的一层。

目前已经有：

- `StateSet` 单状态目录
- `StateArray` / `StateArrayRegistry` 数组状态抽象
- `Port` / `PortGroup`
- `KernelComponent`
- `Kernel`
- `CycleSimulator`
- `SimulationSession`

并且已经在这一层做了较多检查，例如：

- 注册重名
- 元素名/别名冲突
- 端口方向不匹配
- 端口类型/位宽不匹配
- 查找失败
- clone/fullcopy 布局不一致

此外，这一层已经开始接入统一 `Error` 模块，是当前最核心的错误发现层。

### 3. test 标准建模模板层

#### 分工

负责把当前抽象层落成一批“可运行、可观察、可复用、可生成”的标准样板。

典型职责：

- 给标准模块提供手写建模范例
- 给后续自动生成器提供结构模板
- 给 validate 规则提供参考对象
- 给未来导出链路提供“目标表达形式”的样本

#### 当前已做到

目前已经有代表性样例：

- `abctest`
- `led_waterfall`
- `riscv_regfile`

这些 test 已经不只是验证功能，
同时也在逐步形成一种稳定的建模写法：

- 每个标准模块单独目录
- kernel/component 拆分明确
- simulator 输入驱动单独放
- 逐拍输出可读

这一层已经开始发挥“标准模板”的作用，但还没有完全收敛成统一规范。

### 4. Validate 层

#### 分工

负责在真正 `run()` 或导出前，从整体上检查对象图是否合法。

典型职责：

- 是否缺少关键连接
- 是否缺少必需组件
- 标准模块拓扑是否完整
- 某些顺序/协议约束是否满足
- 导入生成出的模型是否满足最低合法性
- 导出前模型是否满足最低后端要求

#### 当前已做到

这一层目前**还没有系统化**。

现状是：

- 很多检查还散落在 elaboration 层
- 主要通过“构造时直接抛错”实现
- 还没有统一的：
  - `Diagnostic`
  - `validate()`
  - `collect_diagnostics()`

所以这一层是当前下一步最值得补的层。

### 5. 数据 / 配置 / 导入描述层

#### 分工

负责给 XS 提供结构描述来源，尤其是未来导入生成链路的描述来源。

典型职责：

- 固定寄存器表
- CSR 表
- 命名表
- XO/HLS 解析结果
- 自动生成所需的中间结构描述

#### 当前已做到

目前这层还是偏轻量的：

- 主要靠 C++ 常量表
- 有 Python 脚本参与解析 XO / HLS 相关内容

但还没有形成一套正式的结构描述协议。

这一层当前是“已感受到需求，但还未正式成型”的状态。

### 6. 静态分析层

#### 分工

负责在编译器之外，用额外分析工具更早发现模式错误。

典型职责：

- API 使用方式检查
- 某些固定写法约束
- 某些结构模式缺失
- 某些模块组织方式违规

#### 当前已做到

当前基本还没做。

也就是说：

- 还没有 `clang-tidy` 规则
- 还没有自定义 `clang` 工具
- 也没有独立的代码模式检查器

这层是未来增强项，不是当前优先级。

### 7. 后端层

#### 分工

负责导入和导出。

一侧是：

- 从 HLS / XO / Vivado 产物还原 XS 周期模型

另一侧是：

- 从 XS 抽象继续导出到 HLS
- 或继续导出到 Verilog / SystemVerilog

#### 当前已做到

当前只做了很早期的一部分铺垫：

- 已经有 Python 脚本去解析 XO/HLS 示例
- test 已经开始按“可生成模板”的方向写
- 抽象层已经开始往中间表示方向收

但真正意义上的后端层，目前还没正式建立。

所以这一层的状态是：

- 方向明确
- 基础铺垫已开始
- 真正实现还未展开

## 结论先行

当前阶段更适合继续打基础，不建议马上把主要精力投入到“编译期验证”。

原因有两类。

### 一类原因：抽象层仍在快速演进

- 很多约束目前仍然是运行时数据
  - 名字
  - 连接关系
  - shape
  - 执行顺序
- 如果现在强推编译期验证，会过早冻结接口
- 模板会迅速变重
- 接口一改就容易返工

### 另一类原因：test 还承担生成链路职责

- test 未来要同时服务“手写验证”和“自动生成周期仿真器”
- 还可能继续服务“导出 HLS / Verilog”
- 这意味着当前抽象更像一种中间表示，而不是单纯测试代码

所以更合理的顺序是：

1. 先把中间表示层做稳
2. 再挑最固定的一小块做编译期验证试点
3. 最后再决定是否扩大编译期约束范围

## 为什么先打基础

当前工程里最关键的问题，不是“能不能再早一点在编译期炸”，而是：

- 错误能不能统一格式
- 诊断能不能集中收集
- 对象图是否合法能不能在 run 前系统检查
- test 能不能收敛成稳定的中间建模模板
- 导入/导出链路以后有没有统一落脚抽象

如果这些基础没稳，编译期化只会导致：

- 模板越来越重
- 灵活性下降
- 接口一改就返工
- 诊断信息反而更难读

## 建议的技术路线

### 1. 编译期静态层

建议语言/实现：

- C++
- 模板
- `static_assert`
- `constexpr`

适合放在这一层的约束：

- 模板参数合法性
- 固定 shape
- 固定数量
- 某些静态禁止组合

例如：

- `State<T*>` 禁止
- 固定寄存器表元素数是否匹配
- 静态数组维度是否合法

这一层当前不建议处理运行时字符串名字冲突。

### 2. Elaboration / 构造层

建议语言/实现：

- 继续使用 C++
- 继续沿用当前对象模型
- 统一通过 `Error` 模块反馈

这一层是当前主战场。

适合放在这一层的约束：

- `StateSet` 注册重名
- `StateArrayRegistry` 注册重名
- `StateArray` 元素名/别名冲突
- 端口方向不匹配
- 端口位宽/类型不匹配
- clone/fullcopy 后布局不一致
- kernel/component/portgroup 查找失败

### 3. Validate 层

建议语言/实现：

- 继续使用 C++
- 自己做轻量级诊断系统

建议新增：

- `struct Diagnostic`
- `enum Severity`
- `collect_diagnostics(...)`
- `validate()`
- `validate_or_throw()`

这一层的目标是：

- 在真正 `run()` 前发现结构问题
- 支持一次性收集多个错误
- 不只抛出第一个错误

适合放在这一层的约束：

- 必需端口是否已连接
- 必需组件是否存在
- 标准模块拓扑是否完整
- 某些阶段顺序是否满足要求
- 某些协议约束是否满足
- 导入生成出的模型是否满足最低合法性
- 导出前模型是否满足后端约束

### 4. 数据 / 配置 / 导入描述层

建议实现：

- 当前阶段：优先用 C++ 常量表
- 中后期：逐步引入 YAML / JSON / TOML 或 Python 结构描述

适合放在这一层的内容：

- 固定寄存器表
- CSR 表
- 标准端口命名表
- 模块标准定义
- XO/HLS 解析结果的结构化描述

如果只是少量固定表，优先用 `constexpr` 常量，不要过早上 DSL。

但要提前承认：

- 未来如果要从 XO / HLS 产物自动还原周期级仿真器
- 那就天然需要一层结构化描述

所以这层未来很可能会变成：

- Python 解析脚本
- 某种结构描述
- 再生成 XS 的 C++ 建模代码

### 5. 静态分析层

如果以后真的想做“像编译器一样更早炸，但又不改编译器”，更适合：

- `clang-tidy`
- `clang libTooling`

不建议现在去滚一个 GCC / g++ 分支。

原因：

- GCC 前端复杂度太高
- 长期维护成本极高
- 升级编译器版本会非常痛苦
- 你现在的需求更像领域约束，不是语言本身改造

### 6. 后端层

当前不建议把后端实现做得太重，但必须在路线图里承认它是目标之一。

因为 test 的双重作用决定了：

- 一条方向是从 HLS/XO 自动导入周期仿真器
- 另一条方向是从 XS 抽象继续导出到 HLS / Verilog

所以后端层最终建议做成：

#### 导入侧

- XO / HLS 产物解析
- 自动生成 XS 周期模型

#### 导出侧

- XS 标准模块到 HLS 的生成
- XS 更低层抽象到 Verilog 的生成

#### 中间共用层

- 导入和导出尽量共用同一份结构抽象

当前阶段没有必要把复杂度提前引入，但要给这条路线留空间。

## 推荐推进顺序

### 第一阶段：继续打基础

优先级最高：

1. 统一错误体系全覆盖
2. `Diagnostic` 与 `validate()` 骨架
3. 标准模块测试继续补齐
4. 把 test 逐步收敛成“标准建模模板”
5. 抽象层命名和语义继续收紧

### 第二阶段：让 test 更像中间表示

重点不是“多写几个测试”，而是让 test 具备模板特征：

- 每个标准模块有稳定的组织方式
- 输入输出定义清晰
- 组件拆分方式稳定
- 便于 Python 脚本自动生成同类结构
- 也便于以后反向导出

### 第三阶段：做局部编译期试点

只挑“最固定的一小块”做静态化，不要全仓库一起推。

最适合试点的对象：

- RISC-V 寄存器组固定接口
- CSR 固定表
- 固定 shape 的数组状态

目标不是全盘替换，而是验证：

- 收益是否明显
- 写法是否还能接受
- 报错是否比运行时更清楚

### 第四阶段：决定是否扩大

只有当试点证明收益足够大时，再考虑扩大到：

- 固定名字 bundle
- 固定端口协议
- 静态名字表

## 对当前工程的具体建议

### 现在立刻值得做的

1. 把剩余 `std::runtime_error` 收进统一 `Error` 模块
2. 给 `Port` / `Kernel` / `Simulator` 建 `validate()` 接口
3. 给标准模块建立“结构验证规则”
4. 把 test 目录里的标准模块写法逐步模板化
5. 约定一套“适合自动生成”的 test / kernel 组织方式
6. 补更多“标准硬件部件”测试
   - register file
   - fifo
   - csr bank
   - simple pipeline
7. 把 Python XO 解析脚本与 XS 模块组织方式逐步对齐

### 现在不建议做的

1. 不建议直接做定制编译器
2. 不建议滚一个自定义 `g++`
3. 不建议全仓库模板化静态名字系统
4. 不建议过早引入复杂 DSL
5. 不建议现在就为了导出 Verilog 把后端 IR 做得过重

## 一句话路线图

当前路线应该是：

- 先把错误、validate 和中间抽象打稳
- 再把 test 收敛成可手写、可自动生成、可继续导出的统一模板
- 再做小范围编译期试点
- 最后决定是否扩大静态验证与导入/导出链路

也就是说：

**先把“中间表示层”打稳，再局部编译期化，而不是反过来。**
