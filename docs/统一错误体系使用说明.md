# 统一错误体系使用说明

## 目标

把当前工程里分散的 `throw std::runtime_error(...)` 收束到一个统一模块中，
为后续做“更早发现问题”“分层报错”“集中诊断”打基础。

## 当前实现

新增文件：

- [src/include/base/Error.h](/home/pyx/ProjectFS/Project-X/Project-XS/src/include/base/Error.h)
- [src/base/Error.cpp](/home/pyx/ProjectFS/Project-X/Project-XS/src/base/Error.cpp)

核心接口：

- `project_xs::sim::error::Stage`
- `project_xs::sim::error::Kind`
- `project_xs::sim::error::make(...)`
- `project_xs::sim::error::raise(...)`

## 错误格式

统一格式：

`[stage][kind][scope] detail`

例如：

```text
[elaboration][duplicate-name][StateSet] duplicate state name: rs1_addr
[elaboration][not-found][Kernel] component not found: write_stage
[runtime][invalid-argument][CycleSimulator] frequency_hz must be > 0
```

## Stage 含义

- `CompileTime`
- `Elaboration`
- `Validate`
- `Runtime`
- `Backend`
- `IO`
- `Data`

## Kind 含义

- `InvalidArgument`
- `NotFound`
- `DuplicateName`
- `LayoutMismatch`
- `TypeMismatch`
- `DirectionMismatch`
- `ConnectionMismatch`
- `ConstraintViolation`
- `Unsupported`
- `IOError`
- `DataFormat`

## 如何使用

### 直接抛错

```cpp
#include "base/Error.h"

project_xs::sim::error::raise(
    project_xs::sim::error::Stage::Elaboration,
    project_xs::sim::error::Kind::DuplicateName,
    "StateSet",
    "duplicate state name: rs1_addr");
```

### 先构造再抛

```cpp
throw project_xs::sim::error::make(
    project_xs::sim::error::Stage::Runtime,
    project_xs::sim::error::Kind::InvalidArgument,
    "SimulationSession",
    "frequency_hz must be > 0");
```

## 推荐约定

### Scope 怎么写

推荐用“对象类型 + 关键信息”：

- `Port`
- `Port connect`
- `PortGroup`
- `Kernel sim_a`
- `KernelComponent write_stage`
- `CycleSimulator simulator_a`
- `StateSet`
- `StateArray gpr`

### Stage 怎么选

- 注册、命名、连线、clone、fullcopy 相关：
  `Elaboration`
- 真正 run 中发现的非法状态：
  `Runtime`
- 文件读取、打开失败：
  `IO`
- 数据格式不对、矩阵文件非法：
  `Data`
- 未来导出 Verilog 失败：
  `Backend`

## 这项工作的意义

这项工作对后续“尽早发现问题”是有利的，原因有三点：

1. 错误先统一格式  
   后面做日志、诊断、IDE 跳转、批量收集时才有基础。

2. 错误先统一阶段  
   后面你想区分“构造期炸”还是“运行期炸”，不需要再重新全仓库梳理。

3. 后续可以平滑扩展  
   现在先是 `throw` 单条异常；
   以后可以扩成：
   - `Diagnostic` 对象
   - 多错误收集
   - `validate()` 聚合输出

## 推荐后续步骤

   把“单点抛错”逐步扩成“集中收集并一次性反馈”
