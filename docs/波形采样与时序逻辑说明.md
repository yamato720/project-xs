# 波形采样与时序逻辑说明

日期：2026-05-14

这份文档说明当前快照/波形采样看到的到底是什么，以及为什么有些波形看起来反常，但仿真逻辑本身仍然成立。

核心结论：

- `State` 只是当前对象里的变量，不自带 `reg/wire` 时序语义。
- `reg/wire` 语义在 `Port` 上，不在 `State` 上。
- session 级采样把一次 `simulator.step()` 当成一个已经完成的动作，因此默认只能看到稳定后的结果。
- 在哪一层打开自动采样，就最清楚地看到哪一层自己的拍内 `reg/wire` 差异；更外层通常只能看到内层完整 `step/run/end_cycle` 后的提交状态。
- `SimulationSession` 自己没有端口组，因此不会产生 session 自身的 `reg/wire` 差异，只能看到下层对象被投影出来的状态。
- 要看 `reg output` 在拍内“先暂存、后提交”的差异，应该看 kernel/component/simulator 自己的自动采样阶段，而不是只看 session 级波形。
- checkpoint 保存端口的 `visible/pending/bound` 完整运行态；HTML 波形默认画的是端口当前 `visible`，不是 `pending`。
- 自动采样是连续波形记录；自动段的第一帧和最后一帧会额外落本地 checkpoint，手动采样则直接保存一个 checkpoint。

## 1. 对象层级

当前仿真对象从外到内大致是：

```text
SimulationSession
  CycleSimulator
    Kernel
      KernelComponent
```

`SimulationSession` 负责多模拟器调度，特别是不同频率的 simulator 如何在同一个 session tick 下推进。`CycleSimulator` 负责自己一个周期里的输入同步、业务推进、输出发射、kernel 推进和拍末提交。`Kernel` 和 `KernelComponent` 再按类似方式管理自己的端口、状态和子组件。

每层都可以打开快照采样：

- `SimulationSession::start_snapshot_capture()`
- `CycleSimulator::start_snapshot_capture()`
- `Kernel::start_snapshot_capture()`
- `KernelComponent::start_snapshot_capture()`

打开在哪一层，波形就以哪一层的视角记录。session 级会把下层 simulator 作为已经调度完成的整体来看；kernel/component 级则能看到该对象自己 `run()` 和 `end_cycle()` 中间的细阶段。

这里有一条很重要的边界：自动采样不会因为在外层打开，就自动插进所有内层对象的每个拍内阶段。比如 session 级采样发生在 `simulator->step()` 外侧，它能记录 simulator/kernel/component 的状态投影，但通常看不到这些内层对象在 `emit_outputs()` 和 `end_cycle()` 之间的短暂差异。要看哪一层的端口 `visible/pending` 变化，就应该在那一层打开自动采样。

## 2. 自动采样点位置总览

自动采样点的位置要按“刚调用某个函数前”还是“某个函数返回后”理解。阶段名里的 `After` 都表示前面的动作已经完成；没有 `After` 的 begin 类阶段通常表示该层本轮工作刚开始、关键动作还没发生。

下面流程里的 `[波形采样: ...]` 就是实际记录到 `waveform.jsonl` / HTML 的时间点。也就是说，某一帧标成哪个 stage，就表示它记录在图中这个标识所在的位置，而不是记录在相邻函数内部。

### 2.1 每次采样到底采到什么

每次 `capture_snapshot(stage)` 都先生成一份完整运行态快照，然后在自动采样打开时追加一帧波形投影。

完整快照用于 checkpoint / restore，内容包括：

- 当前对象的周期、频率、结束标志等运行态。
- 当前对象的 `State` 和 `StateArray` 当前值。
- 当前对象所有 `PortGroup` 的端口运行态。
- 每个端口的 `visible_storage`、`pending_storage`、绑定变量 `bound_storage`、`valid` 和 `pending_valid`。
- 下层对象快照：session 包含 simulator；simulator 包含 kernel；kernel 包含 component。

波形投影用于 `waveform.jsonl` 和 HTML，内容更偏显示：

- 端口信号：端口名、方向、类型、位宽、有效位、显示名、说明、层级 timing 元数据，以及当前 `visible_storage` 格式化后的值。
- state 信号：state 当前值、类型、位宽、显示名、说明和层级 timing 元数据。
- array 信号：数组元素当前值、类型、位宽、显示名、说明和层级 timing 元数据。
- simulator / kernel / component 作为采样根对象时，都会写入自己的 `__cycle` 运行期周期信号。
- session 级会为每个 simulator 写入 `__cycle` 运行期周期信号，以及 simulator 频率、周期比例、finished 状态。

这里最容易误解的一点是：**HTML 当前显示的是波形投影里的 visible 值，不直接显示 pending。** pending 在完整快照/checkpoint 里存在，但没有作为默认波形值画出来。所以 reg output 在 `AfterEmitOutputs` 阶段如果已经写了 pending，HTML 仍会显示旧 visible，这是符合当前波形投影语义的。

### 2.2 自动段边界

#### [波形采样: AutomaticSegmentBegin]

位置：`start_snapshot_capture(Automatic)` 内部，自动段目录创建之后，调用者后续仿真动作之前。

采到的东西：

- 当前目标对象的完整快照。
- 当前目标对象的第一帧波形投影。
- 这一帧通常也是自动段的 `segment_first` checkpoint 来源。

注意：它不是某个周期固定阶段，而是“开始自动采样”这一刻的状态。

---

#### [波形采样: AutomaticSegmentEnd]

位置：`stop_snapshot_capture()` 内部，自动段结束前，写 manifest 和生成 HTML 之前。

采到的东西：

- 当前目标对象的完整快照。
- 当前目标对象的最后一帧波形投影。
- 这一帧会成为自动段的 `segment_last` checkpoint 来源。

注意：它同样不是某个周期固定阶段，而是“停止自动采样”这一刻的状态。

---

### 2.3 SimulationSession

#### [波形采样: SessionStepBegin]

位置：本 session tick 刚开始之后、调度任何 simulator 之前。

采到的东西：

- `SimulationSession` 完整快照：`current_tick`、run time、finished、所有 scheduled simulator 的完整快照。
- 波形投影：session 频率、当前 tick、当前仿真时间、finished。
- 对每个 simulator：本地 cycle、频率、`session_ticks_per_simulator_cycle`、`simulator_cycles_per_session_tick`、finished。
- 对每个 simulator 的信号投影：`__cycle`、端口 visible 值、state 值、array 值，以及下层 kernel/component 的投影。

注意：此时本 tick 内还没有调用任何 `simulator->step()`，所以看到的是上一轮已经稳定的下层状态。

---

#### [波形采样: SessionAfterDispatch]

位置：本 tick 该调度的 simulator 都完整 `step()` 返回之后、`current_tick_++` 之前。

采到的东西：

- session 自己的 `current_tick` 还没加一。
- 已经被调度的 simulator，其完整 `step()` 已经结束，包括其内部 kernel/component 的 `end_cycle()`。
- 未被调度的低频 simulator，仍保持上一帧状态。
- 波形投影仍然只显示下层端口 visible 值，不显示 pending。

注意：session 不插进 `simulator->step()` 内部，所以这里通常看不到下层 `AfterEmitOutputs` 到 `EndCycleEnd` 之间的拍内差异。

---

#### [波形采样: SessionStepEnd]

位置：`current_tick_++` 和 finished 更新之后、`SimulationSession::step()` 返回之前。

采到的东西：

- session tick 已经加一。
- session finished 状态已经根据运行时间和 simulator 完成情况更新。
- 下层 simulator 的端口和 state 通常不会相对 `SessionAfterDispatch` 再变化。

注意：这个点主要用于看 session 自己的时间推进和结束状态。

---

### 2.4 CycleSimulator

#### [波形采样: CycleStepBegin]

位置：本 simulator 周期刚开始之后、同步输入之前。

采到的东西：

- simulator 完整快照：`current_cycle`、频率、finished、自己的 state/array/port，以及全部 kernel/component 快照。
- 波形投影：simulator 自己和下层对象的端口 visible 值、state 值、array 值、层级 timing 元数据。

注意：此时 simulator 自己输入还没同步，输出还没重新发射，看到的是上一拍提交后的状态。

---

#### [波形采样: CycleAfterInputSync]

位置：simulator 自己端口组输入同步之后、`run_single()` 之前。

采到的东西：

- wire input：上游 visible 已经同步到端口 visible 和绑定 state。
- reg input：绑定 state 看到旧 visible；上游新 visible 已采到 pending，但 HTML 仍显示端口 visible。
- simulator 自己业务 state 还没有执行本拍 `run_single()` 更新。

注意：如果怀疑 input reg 的采样方向，要看 checkpoint 里的 pending；HTML 默认只看 visible。

---

#### [波形采样: CycleAfterRunSingle]

位置：simulator 自己业务逻辑完成之后、`emit_outputs()` 之前。

采到的东西：

- simulator 自己的 state 可能已经被 `run_single()` 改写。
- simulator 自己输出端口还没 emit，本帧端口 visible 仍是上一阶段的值。
- 下层 kernel 还没有在本 simulator 周期内运行。

注意：这个点适合看“业务变量是否已经算出来”，不适合看“输出端口是否已经发射出去”。

---

#### [波形采样: CycleAfterEmitOutputs]

位置：simulator 自己输出发射之后、任何 `kernel->run()` 之前。

采到的东西：

- simulator 自己 wire output：HTML 已显示新 visible。
- simulator 自己 reg output：完整快照里 pending 已是新值；HTML 仍显示旧 visible。
- 下层 kernel 仍未运行本 simulator 周期。

注意：这是观察 simulator 自己 output reg/wire 差异的关键点。

---

#### [波形采样: CycleAfterKernelRun]

位置：所有 `kernel->run()` 完成之后、simulator 自己 `port_group->end_cycle()` 之前。

采到的东西：

- 所有 kernel 的 `run()` 已完成。
- kernel 自己输出端口已经 emit，但 kernel `end_cycle()` 还没执行。
- simulator 自己 reg output 也还没提交。
- 完整快照里可以保存各层 pending；HTML 仍显示各端口 visible。

注意：这个点有机会在 simulator 级看到 kernel 运行后、提交前的状态，但不会展开每个 kernel 的内部细阶段。

---

#### [波形采样: CycleAfterPortGroupEndCycle]

位置：simulator 自己端口组 `end_cycle()` 之后、`kernel->end_cycle()` 之前。

采到的东西：

- simulator 自己 reg input/output 已经完成 `pending -> visible`。
- kernel/component 的 reg 端口还没有统一提交。
- HTML 上 simulator 自己 reg output 已经显示新 visible。

注意：这个点主要用于区分 simulator 自己端口提交和下层 kernel/component 提交的先后。

---

#### [波形采样: CycleAfterKernelEndCycle]

位置：所有 kernel/component 的 `end_cycle()` 完成之后、`current_cycle_++` 之前。

采到的东西：

- kernel/component 的 reg 端口已经完成 `pending -> visible`。
- simulator 当前周期号仍是本拍周期号，还没加一。
- HTML 上下层 reg output 通常已经显示提交后的 visible。

注意：如果在这个点才看 reg/wire，拍内差异已经被提交动作消掉了。

---

#### [波形采样: CycleStepEnd]

位置：simulator 周期号已加一、子 kernel 当前周期已同步、finished 状态已更新之后，`step()` 返回之前。

采到的东西：

- simulator `current_cycle` 表示下一次 `step()` 将执行的周期。
- 子 kernel 的当前周期也已经随父 simulator 更新。
- 端口和 state 通常相对 `CycleAfterKernelEndCycle` 不再变化，主要变化是周期元数据。

---

### 2.5 Kernel

#### [波形采样: KernelRunBegin]

位置：`current_cycle_` 已设置之后、同步 kernel 输入之前。

采到的东西：

- kernel 完整快照：当前周期、频率、state/array/port、内部 component 快照。
- 波形投影：kernel 自己和 component 的端口 visible 值、state 值、array 值。

注意：此时 kernel 自己输入还未同步，本轮业务逻辑还没执行。

---

#### [波形采样: KernelAfterInputSync]

位置：kernel 自己端口组输入同步之后、`run_single()` 之前。

采到的东西：

- kernel 自己 wire input 已经把上游 visible 同步到绑定 state。
- kernel 自己 reg input 已经把旧 visible 写入绑定 state，新输入采到 pending。
- component 还没有由 kernel 默认 `run_single()` 推进。

---

#### [波形采样: KernelAfterRunSingle]

位置：kernel 业务逻辑完成之后、`emit_outputs()` 之前。

采到的东西：

- kernel 自己 state 可能已经更新。
- 如果使用默认 `Kernel::run_single()`，内部 component 已经按顺序完成各自 `run()`，但 component `end_cycle()` 还没执行。
- kernel 自己输出端口还没 emit，因此 HTML 上输出端口 visible 仍可能是旧值或 `z`。

注意：这个点适合看 state 是否已经算好，以及 component run 后的投影。

---

#### [波形采样: KernelAfterEmitOutputs]

位置：kernel 自己输出发射之后、`after_outputs_emitted()` 之前。

采到的东西：

- kernel 自己 wire output：HTML 已显示新 visible。
- kernel 自己 reg output：完整快照里 pending 已是新值；HTML 仍显示旧 visible。
- kernel state 显示业务逻辑后的当前值。
- component 仍处于 run 后、end_cycle 前的投影状态。

注意：这是观察 kernel 自己 output reg/wire 差异的关键点。

---

#### [波形采样: KernelRunEnd]

位置：`after_outputs_emitted()` 和 `handle_latency_event()` 完成之后、`Kernel::run()` 返回之前。

采到的东西：

- 输出后钩子和 latency 事件可能造成的 state 变化。
- kernel 自己 reg output 仍未提交，HTML 仍显示旧 visible。
- component 仍未执行统一 `end_cycle()`。

注意：如果派生类没有在输出后钩子里改状态，这个点通常和 `KernelAfterEmitOutputs` 很接近。

---

#### [波形采样: KernelEndCycleEnd]

位置：kernel 自己端口组、内部 component 和 `end_cycle_extra()` 全部完成之后，`Kernel::end_cycle()` 返回之前。

采到的东西：

- kernel 自己 reg input/output 已完成 `pending -> visible`。
- 内部 component 的 reg 端口也已完成 `pending -> visible`。
- HTML 上 kernel/component 的 reg output 通常已经显示提交后的 visible。

注意：到这个点再看 reg/wire，已经是提交后的稳定状态。

---

### 2.6 KernelComponent

#### [波形采样: ComponentRunBegin]

位置：`current_cycle_` 已设置之后、同步 component 输入之前。

采到的东西：

- component 完整快照：当前周期、频率、phase/align、state/array/port。
- 波形投影：component 自己的端口 visible 值、state 值、array 值。

---

#### [波形采样: ComponentAfterInputSync]

位置：component 自己端口组输入同步之后、`run_single()` 之前。

采到的东西：

- component 自己 wire input 已经同步到绑定 state。
- component 自己 reg input 已把旧 visible 写入绑定 state，新输入采到 pending。
- component 业务逻辑尚未执行。

---

#### [波形采样: ComponentAfterRunSingle]

位置：component 业务逻辑完成之后、`emit_outputs()` 之前。

采到的东西：

- component 自己 state 可能已经更新。
- component 输出端口尚未 emit，HTML 上输出端口 visible 仍是旧值或 `z`。

---

#### [波形采样: ComponentAfterEmitOutputs]

位置：component 自己输出发射之后、`write_debug()` / `after_outputs_emitted()` 之前。

采到的东西：

- component wire output：HTML 已显示新 visible。
- component reg output：完整快照里 pending 已是新值；HTML 仍显示旧 visible。
- component state 显示业务逻辑后的当前值。

注意：这是观察 component 自己 output reg/wire 差异的关键点。

---

#### [波形采样: ComponentRunEnd]

位置：`write_debug()`、`after_outputs_emitted()` 和 `advance_phase()` 之后，`KernelComponent::run()` 返回之前。

采到的东西：

- 输出后钩子可能造成的 state 变化。
- phase 已经推进。
- component 自己 reg output 仍未提交，HTML 仍显示旧 visible。

---

#### [波形采样: ComponentEndCycleEnd]

位置：component 自己端口组 `end_cycle()` 和 `end_cycle_extra()` 完成之后，`KernelComponent::end_cycle()` 返回之前。

采到的东西：

- component 自己 reg input/output 已完成 `pending -> visible`。
- HTML 上 component reg output 通常已经显示提交后的 visible。

注意：component 级是观察 component 本地端口时序最细的层级。

## 3. 一个 simulator step 里发生了什么

当前 `CycleSimulator::step()` 不是“只调用一次业务函数”，而是一整拍完整流程：

```text
[波形采样: CycleStepBegin]
sync_inputs()
[波形采样: CycleAfterInputSync]
run_single()
[波形采样: CycleAfterRunSingle]
emit_outputs()
[波形采样: CycleAfterEmitOutputs]
kernel->run(current_cycle_)
[波形采样: CycleAfterKernelRun]
port_group->end_cycle()
[波形采样: CycleAfterPortGroupEndCycle]
kernel->end_cycle()
[波形采样: CycleAfterKernelEndCycle]
current_cycle_++
[波形采样: CycleStepEnd]
```

这些阶段的含义如下。

`CycleStepBegin` 是 simulator 本拍刚开始，输入还没同步，输出也还没重新发射。此时看到的是上一拍已经提交好的可见状态。

`sync_inputs()` 会把 simulator 自己端口组的输入同步到绑定变量。如果是 wire 输入，它会立即读取上游当前可见值；如果是 reg 输入，它先把自己上一拍的可见值写回绑定变量，再把上游当前可见值采到 pending 里，等拍末提交。

`run_single()` 是 simulator 自身的业务逻辑。比如 `abctest` 里的 `DemoCycleSimulator` 会在这里把 `source_out` 设成当前 simulator 周期号。

`emit_outputs()` 会把 simulator 自己绑定 state 的值发射到输出端口。wire 输出在这里立即变成可见值；reg 输出只写 pending，还不会变成 visible。

`kernel->run(current_cycle_)` 会按注册顺序推进每个 kernel。一个 kernel 内部又会执行自己的 `sync_inputs()`、`run_single()`、`emit_outputs()` 等阶段。

`port_group->end_cycle()` 会提交 simulator 自己端口组里的 reg 输入/输出。也就是把 pending 变成 visible。

`kernel->end_cycle()` 会递归提交 kernel 及其内部 component 的端口状态。kernel/component 的 reg output 真正对外可见，通常是在这里之后。

`current_cycle_++` 发生在本拍所有 run 和 end_cycle 都完成之后。因此 `current_cycle()` 表示“下一次 step 将要执行的周期号”，不是“当前正在执行中的周期号”。

## 4. Kernel 和 Component 的拍内顺序

`Kernel::run(cycle)` 的关键顺序是：

```text
[波形采样: KernelRunBegin]
port_group->sync_inputs()
[波形采样: KernelAfterInputSync]
run_single(cycle)
[波形采样: KernelAfterRunSingle]
emit_outputs()
[波形采样: KernelAfterEmitOutputs]
after_outputs_emitted(cycle)
handle_latency_event(cycle)
[波形采样: KernelRunEnd]
```

`Kernel::end_cycle()` 的关键顺序是：

```text
port_group->end_cycle()
component->end_cycle()
end_cycle_extra()
[波形采样: KernelEndCycleEnd]
```

`KernelComponent::run(cycle)` 也类似：

```text
[波形采样: ComponentRunBegin]
port_group->sync_inputs()
[波形采样: ComponentAfterInputSync]
run_single(cycle)
[波形采样: ComponentAfterRunSingle]
emit_outputs()
[波形采样: ComponentAfterEmitOutputs]
after_outputs_emitted(cycle)
advance_phase()
[波形采样: ComponentRunEnd]
```

`KernelComponent::end_cycle()` 最后会提交自己的端口，然后打 `ComponentEndCycleEnd`。

所以，如果要观察一个 kernel 的 reg output 是否真的延后一拍，重点看：

- `KernelAfterRunSingle`
- `KernelAfterEmitOutputs`
- `KernelEndCycleEnd`

如果只看 session 级的 `SessionAfterDispatch` 或 `SessionStepEnd`，这时下层 simulator 的整个 `step()` 已经返回，kernel/component 的 `end_cycle()` 也已经完成，很多拍内差异已经被提交后的稳定状态覆盖。

## 5. Port 的 reg/wire 语义

### WirePort

wire 输出：

- `emit_bound_value()` 立即把绑定变量拷到 `visible_storage`
- 同一拍后续读取这个输出时，可以看到新值

wire 输入：

- `sync_input()` 读取上游输出的 `visible_storage`
- 立即把读到的值写进绑定变量

### RegPort

reg 输出：

- `emit_bound_value()` 只把绑定变量拷到 `pending_storage`
- `visible_storage` 暂时不变
- `end_cycle()` 时才把 pending 提交成 visible

reg 输入：

- `sync_input()` 先把自己的旧 visible 写回绑定变量
- 然后把上游当前 visible 采到 pending
- `end_cycle()` 后，pending 才成为下一拍的 visible

这意味着当前的 `reg input` 更像“一个 input wire 后面接了一级内部输入寄存器”，不是 Verilog 语法里的 `input reg`。Verilog 的 input 端口本身没有 reg 类型；如果要表达边沿采样，通常是 wire input 加内部 previous/current state。这里的 `RegPort` 是框架为了测试和建模提供的端口级时序抽象。

## 6. 为什么 session 级波形会让 reg/wire 看起来一样

`SimulationSession::step()` 的核心是：

```text
SessionStepBegin
for each simulator:
  根据频率累加调度
  需要推进时调用 simulator->step()
SessionAfterDispatch
current_tick_++
SessionStepEnd
```

session 不会插进某个 `simulator.step()` 的中间阶段采样。它只在 dispatch 前后看世界：

- `SessionStepBegin`：本 session tick 开始前的稳定状态
- `SessionAfterDispatch`：本 tick 内该跑的 simulator 都已经完整跑完
- `SessionStepEnd`：session tick 加一后的稳定状态

因此 session 级波形里，`KernelAfterEmitOutputs` 和 `KernelEndCycleEnd` 中间那段差异不会单独出现。对于 kernel 的 reg output，session 往往看到的是已经 end_cycle 后的 visible 值，所以容易觉得 reg 和 wire 没有差异。

另外，`SimulationSession` 本身没有 `PortGroup`。所以 session 级波形不会出现“session 自己的 reg/wire 差异”；它显示的是下层 simulator、kernel、component 的状态投影，以及各 simulator 的运行期周期信号。如果要看某个下层对象自己的拍内端口差异，应该对那个对象单独开采样。

`test/abctest` 现在同时打开两组自动采样：

- session 级：`test/abctest/trace/simulation_session_session/.../waveform.html`
- kernel 级：`test/abctest/trace/kernel_sim_a/.../waveform.html`

session 级适合看多时钟域整体推进；kernel 级适合看端口拍内时序。

## 7. abctest 里的典型现象

`AbcDemoKernel` 里有两组观测输出：

- 默认端口组：`A` 是真正组合 wire output；`B/C` 名字对应 Verilog 的 `wire_B/wire_C`，但源头是内部寄存器 `b_out/c_out`，所以用 reg output 表达 posedge 前旧值、拍末提交新值。
- `reg_outputs` 端口组：`A/B/C`，reg output，绑定的 state 是 `reg_A/reg_B/reg_C`

`run_single()` 里会先从子组件输出读出 `A/B/C`，再把 `reg_A/reg_B/reg_C` 赋成同样的值：

```text
reg_A = A
reg_B = B
reg_C = C
```

所以 `State` 层面的 `A` 和 `reg_A` 一样，是预期行为。A 通路真正的差异不在 state，而在两个输出端口：

- `sim_a_ports/output/A` 是 wire output
- `reg_outputs/output/A` 是 reg output

B/C 通路和 A 不同：Verilog 的 `wire_B/wire_C` 虽然是 wire 类型端口，但它们接的是内部寄存器输出。为了让 XS 的 simulator 级波形和 VCD 顶层一致，`sim_a_ports/output/B` 和 `sim_a_ports/output/C` 在 XS 里也按 reg output 投影；它们在 `AfterEmitOutputs` 看到旧 visible，在 `EndCycleEnd` 之后才看到新值。

kernel 级波形中可以看到差异。例如周期 1 附近的核心变化是：

```text
阶段                    wire端口A   reg端口A   state A   state reg_A
KernelAfterRunSingle    z           0          1         1
KernelAfterEmitOutputs  1           0          1         1
KernelEndCycleEnd       1           1          1         1
```

解释：

- `KernelAfterRunSingle`：业务变量已经算出来，`state A/reg_A` 都是 1，但端口还没 emit，所以 wire 端口还没有本拍新值。
- `KernelAfterEmitOutputs`：wire output 立即可见为 1；reg output 只是写了 pending，visible 仍是上一拍的 0。
- `KernelEndCycleEnd`：reg output 完成提交，visible 也变成 1。

session 级看同一段时，常见结果类似：

```text
阶段                  simulator_a cycle   wire端口A   reg端口A
SessionStepBegin      1                   0          0
SessionAfterDispatch  2                   1          1
SessionStepEnd        2                   1          1
```

这不是 reg 失效，而是 session 看见的已经是 `simulator.step()` 完成后的稳定状态。

### 7.1 和 Verilog VCD 中 posedge 前窗口的对应关系

`abctest` 的 Verilog 版本里，testbench 在每个周期先把 `source_out` 改成新值，然后才执行：

```text
clk = 0; eval; dump
clk = 1; eval; dump
clk = 0; eval; dump
```

所以 VCD 在 `clk` 上升沿前会出现一个事件级窗口：`source_out` 和组合 `wire_A` 已经是新值，但 `reg_A` 还保持上一拍。例如 `cycle = 5` 的 posedge 前可以看到：

```text
source_out = 5
wire_A = 5
reg_A = 4
```

XS session 级波形不会显示这个窗口，因为 `SessionAfterDispatch` / `SessionStepEnd` 采样时 `simulator.step()` 已经完整结束，kernel 的 reg output 已经完成 `end_cycle()` 提交。

XS kernel 级波形里可以在 `KernelAfterEmitOutputs` 看到对应的 A 通路差异：

```text
sim_a.kernel_a.kernel_a_ports.input.in = 5
sim_a.sim_a_ports.output.A = 5
sim_a.reg_outputs.output.A = 4
```

随后到 `KernelEndCycleEnd`，`reg_outputs/output/A` 才提交为 5。也就是说，VCD 的这个“幽灵”不是 XS 周期末结果错误，而是 Verilog 事件级观察点比 session 级 XS 波形更靠前。

`abctest` 的 difftest 现在保留三种检查：

- session 级：`SessionStepEnd` 对 Verilator `cycle_end`，验证周期末稳定值。
- simulator 级：`CycleAfterKernelRun` 对 Verilator `pre_posedge_after_input_drive`，用同一层的 `source_out/wire_A/wire_B/wire_C/reg_A/reg_B/reg_C` 验证 posedge 前顶层可见值。
- kernel 级：`KernelAfterEmitOutputs` 只检查 A 通路，用来说明局部 wire/reg 差异。

这里还有一个容易忽略的点：Verilog 的 `wire_B/wire_C` 虽然是 wire 类型输出端口，但它们连接的是内部寄存器 `b_out/c_out`。因此 XS 里 `sim_a_ports/output/B` 和 `sim_a_ports/output/C` 也必须按 reg output 的可见时序投影，不能直接画本拍组件算出的 next 值；否则 simulator 级 HTML 会比 VCD 早一拍看到 B/C 的变化。

## 8. 多时钟域采样

session 有自己的 `frequency_hz`，每个 simulator 也有自己的 `frequency_hz`。当前调度用累加器处理比例关系：

```text
scheduled.accumulated_hz += simulator.frequency_hz()
while accumulated_hz >= session.frequency_hz():
  accumulated_hz -= session.frequency_hz()
  simulator.step()
```

所以：

- simulator 频率等于 session 频率时，每个 session tick 推进一次。
- simulator 频率低于 session 频率时，多个 session tick 才推进一次。
- 非整数比例也支持，通过累加器在多个 tick 间摊开推进时刻。

`abctest` 当前用于覆盖：

- session：4Hz
- `simulator_a`：4Hz，每个 session tick 推进一次
- `simulator_b`：2Hz，约每 2 个 session tick 推进一次
- `simulator_c`：1.5Hz，在 4Hz session 下是非整数比例，约每 8/3 个 session tick 推进一次

session 级 waveform 会为每个 simulator 输出 `__cycle`，它是运行期周期计数投影，不是 `State`。simulator / kernel / component 单独作为采样根对象时，也会输出自己的根对象 `__cycle`。这条信号用于看对应对象的本地周期何时变化，尤其是多时钟域或非整数比例时。

## 9. HTML 中这些采样如何落到波形上

HTML 的横轴不是严格的物理时间轴，也不是“每个硬件周期一格”的周期轴。当前 HTML 横轴是**采样帧序号轴**：

```text
frame 0, frame 1, frame 2, ...
```

每次写入 `waveform.jsonl` 的一行就是一个 frame。这个 frame 可能是：

- 自动段边界：`AutomaticSegmentBegin` / `AutomaticSegmentEnd`
- session 阶段：`SessionStepBegin` / `SessionAfterDispatch` / `SessionStepEnd`
- simulator 阶段：`CycleStepBegin` / `CycleAfter...` / `CycleStepEnd`
- kernel 阶段：`KernelRunBegin` / `KernelAfter...` / `KernelEndCycleEnd`
- component 阶段：`ComponentRunBegin` / `ComponentAfter...` / `ComponentEndCycleEnd`

HTML 画波形时采用 sample-and-hold 规则：

```text
[frame i, frame i+1) 这一段显示 frame i 采到的值
```

也就是说，一段框里的值指的是**这段左边那个采样帧的值**，不是右边结束点的值。右边界只是下一帧到来、值可能发生变化的位置。

### 9.1 HTML 里值的意义

HTML 里每个信号框的值，都是某个 frame 中该信号的波形投影值。它不是重新从 checkpoint 里计算的，也不是把一个周期结束时的值回填到整个周期。

对不同信号类型，值的含义如下：

- `PortInput` / `PortOutput`：端口当前 `visible_storage` 的格式化结果。`reg` 端口的 `pending_storage` 不会默认画出来。
- `State`：该 frame 中绑定 C++ state 的当前值。
- `StateArrayElement`：该 frame 中数组元素的当前值。
- `RuntimeCycle` / `__cycle`：该 frame 中对应对象的当前周期号，是运行期元数据投影，不是 `State`。
- `z`：该端口在这一帧 `valid=false`，表示没有当前可见有效值。

因此，HTML 中某个 `reg output` 框显示的值表示“这一帧外部可见的输出值”，不是“这一帧已经采到 pending、下一拍会提交的值”。如果在 `KernelAfterEmitOutputs` 阶段看到：

```text
state reg_A = 1
reg_outputs/output/A = 0
```

含义是：

- 绑定 state `reg_A` 已经在本拍业务逻辑里变成 1。
- reg output 已经把 1 采进 pending。
- 但 HTML 默认显示的是端口 visible，所以仍显示上一拍可见的 0。

左侧“当前值”栏显示的是光标所在采样点对应 frame 的投影值。它不是某个 `RuntimeCycle` 框结束时的汇总值，也不是从 checkpoint 里重新计算出来的值。

波形区域仍然按逐帧 sample-and-hold 显示：

```text
frame N 的值   显示在 [N, N+1) 区间
frame N+1 的值 显示在 [N+1, N+2) 区间
```

但当前值栏把采样帧边界归给右侧已经生效的 frame。也就是说，当光标线正好落在 `N+1` 这个边界上时，左侧“当前值”栏显示 `frame N+1`，而不是 `frame N`。

这个规则更适合读“周期末尾”的值。比如 reg output 在 cycle 0 开始前可能是 `z`，但在 cycle 0 的 `KernelEndCycleEnd` / `SessionStepEnd` 采样点已经完成提交；光标放在这个周期末边界时，当前值栏应该显示提交后的值，而不是上一段提交前的 `z`。

### 9.2 RuntimeCycle 是普通波形信号

`RuntimeCycle` / `__cycle` 在 HTML 里不是特殊横轴刻度，而是一条普通 bus 风格信号。它的值来自对应对象在该 frame 里记录到的当前周期号：

- session 级：每个 simulator 有自己的 `simulator_name.__cycle`
- simulator 级：根对象的 `current_cycle`
- kernel/component 级：根对象自己的 `current_cycle`

因此，一段 `__cycle = N` 的框表示：

```text
从这段左边的 frame 开始，采样记录里的 current_cycle 是 N；
直到下一帧采到不同 cycle 或 trace 结束。
```

它不表示“这一整段内部正在执行 cycle N 的所有动作”。它只是把多个连续 frame 里相同的 cycle 值合成一个显示框。

### 9.3 多个阶段为什么会落在同一个 RuntimeCycle 框里

同一个周期内会有多个自动采样阶段，而这些阶段的 `current_cycle` 可能相同。因此 HTML 上它们会落在同一个 `RuntimeCycle` 框里。

以 simulator 级为例，假设本轮执行的是 `current_cycle = N`：

```text
[frame a] CycleStepBegin              __cycle = N
[frame b] CycleAfterInputSync         __cycle = N
[frame c] CycleAfterRunSingle         __cycle = N
[frame d] CycleAfterEmitOutputs       __cycle = N
[frame e] CycleAfterKernelRun         __cycle = N
[frame f] CycleAfterPortGroupEndCycle __cycle = N
[frame g] CycleAfterKernelEndCycle    __cycle = N
current_cycle_++
[frame h] CycleStepEnd                __cycle = N + 1
```

HTML 会把 `frame a` 到 `frame h` 之前的这些 frame 合并显示在同一个 `__cycle = N` 框里。也就是说，`__cycle = N` 框内可能包含：

- 周期开始状态
- 输入同步后状态
- 业务逻辑后状态
- 输出 emit 后状态
- kernel run 后状态
- 端口 end_cycle 后状态

它们虽然在同一个 `RuntimeCycle` 框里，但仍是不同 frame。光标落到不同 frame 时，底部状态栏里的 `阶段=...` 会不同，对应信号值也可能不同。

### 9.4 session 级的 RuntimeCycle 更容易让人误读

session 级波形里，`__cycle` 是每个 simulator 的本地周期投影。一次 `SimulationSession::step()` 里，session 自己通常会采：

```text
SessionStepBegin      current_tick = T
SessionAfterDispatch  current_tick = T
SessionStepEnd        current_tick = T + 1
```

而某个 simulator 可能在 `SessionStepBegin` 到 `SessionAfterDispatch` 之间完整执行了一个或多个 `simulator->step()`。所以 session 级 HTML 里：

- `SessionStepBegin` 看到的是调度前的 simulator cycle。
- `SessionAfterDispatch` 看到的是调度后的 simulator cycle。
- `SessionStepEnd` 通常只体现 session tick 已经加一。

如果 simulator 频率低于 session，某些 session tick 内它不会被调度，那么该 simulator 的 `__cycle` 会跨多个 session frame 保持不变。

如果 simulator 频率高于 session，单个 session tick 内可能调用多次 `simulator->step()`。session 级采样仍只会在 dispatch 前后各看一次，因此中间多个 simulator 周期不会在 session 级 HTML 里展开；要展开只能打开 simulator 级采样。

### 9.5 边界帧和普通波形帧

`AutomaticSegmentBegin` 和 `AutomaticSegmentEnd` 目前也会写进 `waveform.jsonl`，所以 HTML 会把它们当 frame 画出来。

从波形阅读语义上，它们更像检查点边界：

- `AutomaticSegmentBegin`：自动段开始时的检查点帧。
- `AutomaticSegmentEnd`：自动段结束时的检查点帧。

它们不是某个仿真函数内部的固定阶段。如果只想读“仿真周期内发生了什么”，重点看 `Session...`、`Cycle...`、`Kernel...`、`Component...` 这些运行阶段。边界帧可以看作段起点/终点标记，不应理解成额外执行了一个周期。

## 10. 从 HLS 仿真视角怎么读这张图

如果目标是仿真一个 HLS 写出来的 kernel，那么这张波形图可以分成两种读法：

- session 级：读 host / testbench 在 kernel 外部能看到什么。
- simulator / kernel / component 级：读 kernel 内部某个周期阶段发生了什么。

### 10.1 周期数看哪个

不要把 `session.current_tick` 直接当成所有 HLS kernel 的周期数。session tick 是外层调度时间。

某个 HLS kernel 真正经历了多少周期，要看它所属 simulator 的周期：

```text
HLS kernel 周期数
= 承载它的 CycleSimulator::current_cycle()
= session 级 HTML 里的 simulator_name.__cycle
```

只有当 session 频率和该 simulator 频率一致，并且每个 session tick 都正好推进一次 simulator 时，session tick 才和这个 HLS kernel 周期数一致。

多时钟域下要分别看：

```text
simulator_a.__cycle -> simulator_a 下面 kernel 的周期数
simulator_b.__cycle -> simulator_b 下面 kernel 的周期数
simulator_c.__cycle -> simulator_c 下面 kernel 的周期数
```

### 10.2 session 级数值表示 host 能看到的值

从 HLS 外部接口视角看，host / testbench 能读到的是端口当前可见值，也就是 `visible_storage`。

因此，在 session 级波形里，一个 simulator/kernel 对外端口的值可以理解为：

```text
在这一帧采样时，host 从该 kernel 外部端口读到的稳定 visible 值。
```

对 `wire output`，这个 visible 通常在该层 `emit_outputs()` 后立即变成新值。

对 `reg output`，只有 `end_cycle()` 把 pending 提交成 visible 后，host 才能读到新值。pending 表示“下一次提交将可见的值”，不表示 host 已经读到了。

所以 session 级波形适合回答：

```text
这个 HLS kernel 跑到所属 simulator 的第 N 个周期边界后，
host / testbench 能从它的外部端口看到什么？
```

### 10.3 一个 simulator 周期框里的值怎么解释

在 HTML 上，`simulator_x.__cycle = N` 的框表示连续多个 frame 里，`simulator_x.current_cycle()` 的投影值是 N。

这个框内的普通信号值仍然要按 frame 读：

```text
[frame i, frame i+1) 显示 frame i 采到的 visible/state 值
```

如果你只关心 HLS 外部行为，优先看 session 级的 `SessionAfterDispatch` 或 `SessionStepEnd` 附近：

- `SessionStepBegin`：本次 session 调度前，host 看到的旧稳定值。
- `SessionAfterDispatch`：本次该跑的 simulator 完整 step 后，host 看到的新稳定值。
- `SessionStepEnd`：session tick 已加一，通常和 `SessionAfterDispatch` 的外部端口值一致。

如果 `simulator_x.__cycle` 从 N 变成 N+1，表示这个 simulator 已经完成了第 N 个 step。此时 session 级波形中对应 kernel 外部端口的 visible 值，可以视为 host 在第 N 个 simulator 周期提交后能读到的结果。

### 10.4 哪些值不要当成 host 读数

以下内容不应该直接当作 host 从 HLS kernel 外部读到的值：

- kernel/component 级 `AfterEmitOutputs` 中 reg output 的 pending 值；HTML 默认也不画 pending。
- `State` 信号，除非这个 state 通过端口发射成外部 visible 值。
- component 内部端口值，除非它就是你要调试的内部连线。
- `AutomaticSegmentBegin` / `AutomaticSegmentEnd` 边界帧，它们主要是检查点边界。

`State` 很适合看“内部变量什么时候算出来”，但 HLS 外部读数要看对外 `PortOutput` 的 visible。

### 10.5 什么时候需要看更内层

session 级适合看：

- host / testbench 可见输出。
- 多 simulator 的周期推进关系。
- kernel 外部接口的最终稳定行为。

simulator 级适合看：

- simulator 自己端口组的输入同步、业务逻辑、输出发射、端口提交顺序。
- session 看不到的 simulator 内部阶段。

kernel 级适合看：

- kernel 自己 `run_single()` 后 state 是否已算出。
- kernel 输出端口 `emit_outputs()` 后，wire 和 reg 的 visible/pending 差异。
- kernel `end_cycle()` 后对外 visible 如何提交。

component 级适合看：

- component 自己的 phase、latency、输入同步和输出发射。
- component 内部 reg/wire 在拍内的差异。

简化成一句话：

```text
看 HLS 对外行为，信 session 级的对应 simulator.__cycle 和输出端口 visible。
看 HLS 内部时序，切到 simulator/kernel/component 级看具体阶段。
```

## 11. 快照记录和文件结构

快照采样有两种模式：

- `Manual`：只在显式调用 `capture_snapshot()` 时记录，产物是一个 checkpoint。
- `Automatic`：在对象自己的关键阶段自动连续记录，产物是一个 segment。

自动段会生成：

```text
trace/<object_kind>_<object_name>/<object_kind>.<date>.<start_cycle>/
  manifest.json
  waveform.jsonl
  waveform.html
  checkpoints/
    first.pxsckpt
    last.pxsckpt
```

`waveform.jsonl` 是逐帧记录，适合 Python/HTML 读取；`waveform.html` 是当前脚本生成的交互查看器；`first.pxsckpt` 和 `last.pxsckpt` 分别对应自动段第一帧和最后一帧，可用于检查和恢复。

恢复 checkpoint 时，目标对象必须已经按同构结构构造完成，也就是名字、端口组、state、component/kernel 数量和顺序要匹配。checkpoint 保存运行态，不保存重新构造整个拓扑的 C++ 代码。

## 12. 调试时应该看哪一层

如果要看整体功能是否按周期前进，优先看 session 级波形。它适合判断多 simulator 的频率、周期推进、全局输入输出和跨模拟器结果是否正确。

如果要看某个 simulator 自身的输入同步、业务逻辑、输出发射和 kernel 执行顺序，打开 simulator 级采样。

如果要看 kernel 端口的 reg/wire 差异，尤其是 `emit_outputs()` 和 `end_cycle()` 之间的差异，打开 kernel 级采样。

如果要看 component 内部 phase、latency、输入同步和输出发射，打开 component 级采样。

如果一个 state 看起来“没有延迟”，先确认你看的是否是 state 本身。state 没有 reg/wire 语义；端口的 visible/pending 才体现 reg/wire 差异。

如果一个 session 级波形里 reg/wire 看起来同时变化，先确认是否已经跨过了下层 `end_cycle()`。session 级采样看不到下层 `simulator.step()` 中间的每个细阶段。

一句话规则是：在哪一层采样，就优先相信那一层自己的阶段；外层波形适合看内层提交后的整体结果，不适合判断内层端口在拍内是否曾经进入 pending 状态。

## 13. 目前波形反常感的主要来源

当前最容易误判的点有三个。

第一，`state reg_A` 这个名字里虽然有 `reg`，但它仍然只是普通 state。它和 `state A` 同拍被赋值，因此它们一样并不说明 reg output 没有延迟。

第二，session 级采样是“包在 simulator 外面”的观察点。它不插入 kernel 的 `emit_outputs()` 和 `end_cycle()` 之间，所以看起来会更接近提交后的数字电路寄存器边界状态。

第三，端口输入和输出的 reg 语义不同于 Verilog 端口声明语法。这里的 `RegPort` 是仿真框架的端口级缓存/提交模型，用来表达跨拍可见性，而不是在复刻 Verilog 的 `input/output reg` 语法。

因此，当前波形“反常”的主要原因是观察层级和观察阶段不匹配，不是端口提交逻辑失效。对照 kernel 级波形看，`KernelAfterEmitOutputs` 到 `KernelEndCycleEnd` 之间能看到 reg output 的 pending/visible 差异。
